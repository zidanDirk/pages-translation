想象一个个人 AI Agent——Email-Bot，旨在帮助你管理收件箱。为了提供价值并有效运作，Email-Bot 可能需要：

- 访问未读邮件内容，从各发件人处获取有用摘要
- 阅读现有邮件以跟踪重要更新、提醒或上下文
- 代表你发送回复或跟进邮件

虽然自动化邮件助手可以提供很大帮助，但这个假设的机器人也展示了 AI agent 如何引入新型风险。值得注意的是，该行业面临的最大挑战之一是 agent 对 **prompt injection** 的敏感性。

**Prompt injection** 是所有 LLM 的一个根本性、尚未解决的弱点。通过 prompt injection，某些类型的不可信字符串或数据片段——当传入 AI agent 的上下文窗口时——可能导致意外后果，例如忽略开发者提供的指令和安全指南，或执行未授权的任务。这一漏洞足以让攻击者控制 agent 并伤害 AI agent 的用户。

以 Email-Bot 为例，如果攻击者在发送给目标用户的电子邮件中放置 prompt injection 字符串，一旦该邮件被处理，他们就可以劫持 AI agent。示例攻击可能包括窃取敏感数据（如私人邮件内容），或执行不必要操作（如向目标用户的好友发送钓鱼消息）。

与许多行业同行一样，我们对 agentic AI 改善人们生活和提高生产力的潜力感到兴奋。实现这一愿景的路径涉及为 AI agent（如 Email-Bot）授予更多能力，包括访问：

- **由未知方创建的数据源**，如入站邮件或从互联网查询的内容
- **私有或敏感数据**，agent 被允许使用这些数据来帮助规划并实现更高个性化
- **可被自主调用的工具**，代表用户完成事务

在 Meta，我们深入思考 agent 如何在平衡这一产品愿景所需的实用性和灵活性的同时，最大程度减少 prompt injection 带来的不良后果，如私人数据泄露、强制代表用户执行操作或系统中断。为了从这一已知风险中最佳保护人们和我们的系统，我们开发了 **Agents Rule of Two**。遵循此框架时，安全风险的严重程度会被确定性降低。

受为 Chromium 开发的同名策略以及 Simon Willison 的"lethal trifecta"启发，我们的框架旨在帮助开发者理解和驾驭当今这些强大的 agent 框架中存在的权衡。

## Agents Rule of Two

从高层次看，Agents Rule of Two 指出：在稳健性研究允许我们可靠地检测和拒绝 prompt injection 之前，agent 在一个会话中必须满足以下三个属性中的**不超过两个**，以避免 prompt injection 的最高影响后果。

- **[A]** agent 可以处理不可信输入
- **[B]** agent 可以访问敏感系统或私有数据
- **[C]** agent 可以更改状态或对外通信

三个属性都可能需要满足以执行请求。如果 agent 需要全部三个属性而不启动新会话（即使用新的上下文窗口），则不应允许 agent 自主运作，至少需要监督——通过 human-in-the-loop 批准或其他可靠验证方式。

## Agents Rule of Two 如何阻止攻击

让我们回到 Email-Bot 示例，看看应用 Agents Rule of Two 如何防止数据泄露攻击。

**攻击场景：** 垃圾邮件中的 prompt injection 包含一个字符串，指示用户的 Email-Bot 收集用户收件箱的私人内容，并通过调用 Send-New-Email 工具将其转发给攻击者。

此攻击成功是因为：

- [A] agent 可以访问不可信数据（垃圾邮件）
- [B] agent 可以访问用户的私有数据（收件箱）
- [C] agent 可以对外通信（通过发送新邮件）

有了 Agents Rule of Two，可以通过几种不同方式防止此攻击：

在 **[BC]** 配置中，agent 只能处理来自可信发件人（如密友）的邮件，防止初始 prompt injection 有效载荷进入 agent 的上下文窗口。

在 **[AC]** 配置中，agent 不会访问任何敏感数据或系统（例如在训练环境中运行），因此任何到达 agent 的 prompt injection 都不会产生有意义的影响。

在 **[AB]** 配置中，agent 只能向可信收件人发送新邮件，或者在人工验证草稿消息内容后发送，防止攻击者最终完成其攻击链。

通过 Agents Rule of Two，agent 开发者可以比较不同设计及其相关的权衡（如用户摩擦或能力限制），以确定哪种选择最适合其用户的需求。

## Agents Rule of Two 的假设示例与实现

让我们看另外三个假设的 agent 用例，了解它们如何选择满足此框架。

### 旅游代理助手 [AB]

这是一个面向公众的旅游助手，可以回答问题并代表用户行事。

- 需要搜索网络以获取最新旅行目的地信息 **[A]**
- 有权访问用户私有信息以完成预订和购买体验 **[B]**

为了满足 Agents Rule of Two，我们通过以下方式对其工具和通信 **[C]** 放置预防性控制：
- 请求人工确认任何操作，如预订或支付定金
- 将网络请求限制为仅来自可信来源的 URL，而不是访问 agent 构建的 URL

### 网页浏览研究助手 [AC]

这个 agent 可以代表用户与网页浏览器交互进行研究。

- 需要填写表单并向任意 URL 发送大量请求 **[C]**
- 必须处理结果 **[A]** 以根据需要重新规划

为了满足 Agents Rule of Two，我们通过以下方式对其访问敏感系统和私有数据 **[B]** 放置预防性控制：
- 在限制性沙箱中运行浏览器，不预加载会话数据
- 限制 agent 访问私有信息（除初始 prompt 外），并告知用户其数据可能如何共享

### 高速内部编码器 [BC]

这个 agent 可以通过在整个组织的内部基础设施上生成和执行代码来解决工程问题。

- 为了解决有意义的问题，必须访问生产系统的子集 **[B]**
- 有能力对这些系统进行状态更改 **[C]**
- 虽然 human-in-the-loop 可以是重要的纵深防御，但开发者旨在通过最小化人工干预来实现规模化运营

为了满足 Agents Rule of Two，我们通过以下方式对任何不可信数据源 **[A]** 放置预防性控制：
- 使用作者谱系来过滤 agent 上下文窗口内处理的所有数据源
- 提供人工审查流程以标记误报，并使 agent 能够访问数据

---

作为通用框架，具体细节至关重要。为了启用更多用例，agent 在同一会话中从一种 Agents Rule of Two 配置安全转换到另一种是可能的。一个具体示例是在 **[AC]** 中开始访问互联网，然后在访问内部系统时通过禁用通信完成到 **[B]** 的单向转换。

虽然为简洁起见省略了完成此操作的所有具体方法，但读者可以通过关注破坏攻击路径——即阻止攻击完成从 **[A] → [B] → [C]** 的完整链条——来推断何时可以安全完成。

## 局限性

需要注意的是，满足 Agents Rule of Two 不应被视为足以防范 agent 其他常见威胁向量（如攻击者提升、垃圾信息扩散、agent 错误、幻觉、过度权限等）或 prompt injection 的较低后果（如 agent 响应中的错误信息）。

同样，应用 Agents Rule of Two 不应被视为减轻风险的终点。满足 Agents Rule of Two 的设计仍可能发生故障（如用户盲目确认警告弹窗），纵深防御是减轻最高风险场景的关键组成部分，当单一层的故障可能是很可能的时候。Agents Rule of Two 是补充——而非替代——如最小权限等常见安全原则。

## 下一步

我们相信 Agents Rule of Two 对今天的开发者是一个有用的框架。我们也对其在规模化安全开发方面的潜力感到兴奋。

随着通过 Model Context Protocol (MCP) 等协议采用即插即用 agentic 工具调用，我们既看到新兴风险也看到机会。虽然盲目将 agent 连接到新工具可能是灾难的秘诀，但通过内置 Rule of Two 意识实现默认安全是可能的。例如，通过在支持的工具调用中声明 Agents Rule of Two 配置，开发者可以更有信心地知道操作将根据其策略成功、失败或请求额外批准。

我们还知道，随着 agent 变得更有用和能力增长，一些高度寻求的用例将难以干净地符合 Agents Rule of Two，如 human-in-the-loop 具有破坏性或无效的后台进程。虽然我们相信传统的软件护栏和人工批准继续是满足当前用例中 Agents Rule of Two 的首选方法，但我们将继续通过对齐控制追求满足 Agents Rule of Two 的监督批准检查，如 oversight agents 和开源 LlamaFirewall 平台。期待在未来分享更多。
